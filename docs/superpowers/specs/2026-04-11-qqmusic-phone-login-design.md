# QQ Music 手机验证码登录设计

## 目标

在现有 QQ 音乐登录弹窗中新增中国大陆手机号 `+86` 验证码登录方式，和现有扫码登录共用同一对话框、同一凭证保存逻辑、同一登录成功回调。

成功标准：

- 用户可在现有 QQ 音乐登录弹窗中切换到手机验证码登录模式
- 用户可输入大陆手机号并发送验证码
- 用户可输入验证码完成登录并保存凭证
- 登录错误在弹窗内联展示，允许直接重试
- 登录成功后无需重启，当前在线页立即刷新为最新登录态

非目标：

- 不支持非 `+86` 区号
- 不支持图形验证码流程的完整承接，只做错误提示
- 不新增独立登录页面或二级弹窗

## 入口与交互

入口保持不变：

- 设置页中的 QQ 音乐登录按钮
- 在线音乐页中的登录按钮

登录弹窗改为单一容器内的两种模式：

- `扫码登录`
- `手机验证码登录`

扫码登录保留现有行为。

手机验证码登录模式包含：

- 固定国家区号标签 `+86`
- 手机号输入框
- 发送验证码按钮
- 验证码输入框
- 登录按钮
- 内联状态/错误提示标签

交互规则：

- 发送验证码前要求手机号格式合法
- 验证码发送成功后保留手机号，允许继续输入验证码
- 登录失败不清空已填内容，允许用户修正后重试
- 成功后沿用现有凭证保存、昵称刷新、页面刷新逻辑

## 客户端接口

在 `plugins/builtin/qqmusic/lib/qqmusic_client.py` 增加两个同步接口。

### `send_phone_auth_code(phone: str, country_code: int = 86) -> dict`

对齐参考实现：

- `module`: `music.login.LoginServer`
- `method`: `SendPhoneAuthCode`
- `param`:
  - `tmeAppid: "qqmusic"`
  - `phoneNo: <手机号字符串>`
  - `areaCode: <国家区号字符串>`

公共参数要求：

- `tmeLoginMethod = 3`

行为：

- 成功返回原始数据字典
- 频率限制、需要验证等错误码由 UI 层转译成人类可读消息

### `phone_authorize(phone: str, auth_code: str, country_code: int = 86) -> dict | None`

对齐参考实现：

- `module`: `music.login.LoginServer`
- `method`: `Login`
- `param`:
  - `code: <验证码字符串>`
  - `phoneNo: <手机号字符串>`
  - `areaCode: <国家区号字符串>`
  - `loginMode: 1`

公共参数要求：

- `tmeLoginMethod = 3`
- `tmeLoginType = 0`

行为：

- 成功返回完整凭证字典
- 返回值兼容现有二维码登录保存逻辑
- 字段归一化沿用当前项目约定，如 `encryptUin -> encrypt_uin`、`loginType -> login_type`

## UI 结构调整

在 `plugins/builtin/qqmusic/lib/login_dialog.py` 中扩展现有 `QQMusicLoginDialog`：

- 新增登录模式切换控件
- 保留二维码模式现有线程和状态流
- 新增手机验证码模式表单区
- 将“状态提示”拆分为：
  - 扫码模式状态标签
  - 手机模式内联提示标签

实现上避免引入第二个对话框；同一个 dialog 内部只切换内容区域显示。

## 状态与错误处理

手机登录错误统一在弹窗内联提示，不弹额外消息框。

错误码映射：

- `CAPTCHA` 类错误：提示“当前登录需要额外验证，请稍后改用扫码登录或稍后重试”
- `FREQUENCY` 类错误：提示“操作过于频繁，请稍后再试”
- `20271`：提示“验证码错误或已失效，请重新输入或重新获取”
- `20274`：提示“当前账号达到设备数量限制，无法继续登录”
- 其他错误：提示“登录失败”并附带 code 或异常信息

输入校验：

- 手机号必须为 11 位数字
- 验证码必须为 4 到 6 位数字

## 成功路径

手机验证码登录成功后：

1. 将凭证保存到插件设置
2. 复用现有昵称刷新逻辑
3. 发射 `credentials_obtained`
4. 由在线音乐页使用新凭证立即刷新 `_qqmusic_service`
5. 收藏、关注歌手等登录态数据无需重启即可重新加载

## 测试策略

优先采用 TDD。

服务层测试：

- `send_phone_auth_code()` 的请求 payload 正确
- `phone_authorize()` 的请求 payload 正确
- `phone_authorize()` 成功后凭证字段正确归一化

UI 测试：

- 登录弹窗可在扫码和手机模式间切换
- 手机号非法时不发送请求并显示内联错误
- 发送验证码成功后更新内联提示
- 登录失败时保留输入并显示内联错误
- 登录成功时发射 `credentials_obtained`

回归测试：

- 现有扫码登录测试继续通过
- 现有在线页登录成功刷新测试继续通过

## 风险与约束

- 当前客户端请求公共参数构建偏 Web 形状，手机登录接口虽然对齐参考实现，但仍需通过测试防止请求形状退化
- 若 QQ 音乐后续增强图形验证码要求，本次实现只能提示用户改用扫码或稍后再试
- 同一弹窗承担两种登录模式后，代码复杂度会上升，需通过小函数拆分控制体积
