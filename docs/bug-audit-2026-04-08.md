# 代码审查缺陷记录（2026-04-08）

本次结论来自代码审查加最小复现脚本，不是“猜测性风险清单”。下面只记录我已经确认的问题。

## 1. 插件 ZIP 安装存在 Zip Slip 路径穿越风险

- 位置：`system/plugins/installer.py:107-117`
- 关键代码：`ZipFile.extractall(extract_root)` 直接解压，没有校验成员路径是否逃逸出 `extract_root`。
- 影响：
  - 恶意插件包可以写出临时目录，覆盖用户工作区或应用数据目录下的任意文件。
  - 这个问题发生在安装阶段，甚至早于 manifest 校验和 import 审计。
- 结论依据：
  - [system/plugins/installer.py](/home/harold/workspace/music-player/system/plugins/installer.py#L107)

## 2. 一个损坏的插件 manifest 会让整个插件发现流程直接崩掉

- 位置：`system/plugins/manager.py:108-145`，`packages/harmony-plugin-api/src/harmony_plugin_api/manifest.py:51-99`
- 根因：
  - `discover_roots()` 对每个插件目录直接 `read_manifest()`。
  - `PluginManifest.from_dict()` 对缺字段/错字段直接抛异常。
  - `discover_roots()`、`list_plugins()`、`load_enabled_plugins()` 都没有在发现阶段隔离单个坏插件。
- 影响：
  - 只要外部插件目录里存在一个损坏的 `plugin.json`，插件列表、插件启动、启停操作都可能整体失败。
- 我这边的最小复现：
  - 放一个正常 builtin 插件和一个缺字段 external 插件后，`manager.discover_roots()` 直接抛 `PluginManifestError: Missing manifest keys...`
- 结论依据：
  - [system/plugins/manager.py](/home/harold/workspace/music-player/system/plugins/manager.py#L108)
  - [manifest.py](/home/harold/workspace/music-player/packages/harmony-plugin-api/src/harmony_plugin_api/manifest.py#L51)

## 3. 在线歌曲缓存删除只删根目录，删不到 provider 子目录里的真实文件

- 位置：`services/download/online_download_gateway.py:177-188`
- 根因：
  - 下载缓存已经按 provider 分目录存放，见 `get_cached_path(..., provider_id=...)`。
  - `delete_cached_file()` 却只遍历 `self._download_dir/<song_mid>.*`，完全不检查 `qqmusic/`、`netease/` 这类子目录。
- 影响：
  - 重新下载/清缓存后，provider 命名空间里的旧文件仍然残留。
  - UI 会认为缓存被清掉了，但磁盘实际没删，后续状态会越来越混乱。
- 我这边的最小复现：
  - 创建 `tmp/qqmusic/song.flac` 后调用 `delete_cached_file("song")`，返回值是 `False`，文件仍然存在。
- 结论依据：
  - [online_download_gateway.py](/home/harold/workspace/music-player/services/download/online_download_gateway.py#L177)

## 4. 云盘文件夹变空后，旧缓存不会被清掉，界面会继续显示“幽灵文件”

- 位置：
  - `repositories/cloud_repository.py:454-502`
  - `ui/views/cloud/cloud_drive_view.py:1136-1144`
- 根因：
  - `cache_files()` 对空列表直接 `return True`，不会删除该目录的旧缓存。
  - `_load_files()` 只有在 `files and len(files) > 0` 时才调用 `cache_files()`，远端目录为空时根本不会刷新本地缓存。
- 影响：
  - 云盘目录删除完文件后，UI 仍然显示旧文件。
  - 用户可能继续对不存在的文件执行播放、下载、删除等操作。
- 我这边的最小复现：
  - 先在 `folderA` 写入缓存文件，再执行 `cache_files(account_id, [])`，`get_files_by_parent(account_id, "folderA")` 仍然返回旧文件 `['f1']`。
- 结论依据：
  - [cloud_repository.py](/home/harold/workspace/music-player/repositories/cloud_repository.py#L454)
  - [cloud_drive_view.py](/home/harold/workspace/music-player/ui/views/cloud/cloud_drive_view.py#L1136)

## 5. 多在线源共用同一个 track id 时，收藏功能会发生 provider 冲突

- 位置：
  - `repositories/favorite_repository.py:23-125`
  - `infrastructure/database/sqlite_manager.py:1051-1060`
  - `system/plugins/plugin_sdk_runtime.py:103-112`
- 根因：
  - 收藏表和收藏查询只按 `cloud_file_id` 去重/查找/删除，没有把 `online_provider_id` 纳入键。
  - 插件运行时的 `remove_library_favorite_by_mid()` 也是只按 `mid` 查，不带 provider。
- 影响：
  - 两个 provider 只要 track id 相同，就不能分别收藏。
  - 取消一个 provider 的收藏，可能误删另一个 provider 的收藏记录或错误命中错误曲目。
- 我这边的最小复现：
  - 插入 `qqmusic:42` 和 `netease:42` 两条在线曲目后：
    - `add_favorite(cloud_file_id='42')` 第一次返回 `True`
    - 第二次返回 `False`
  - 说明系统把两个 provider 的不同歌曲当成同一首歌。
- 结论依据：
  - [favorite_repository.py](/home/harold/workspace/music-player/repositories/favorite_repository.py#L23)
  - [sqlite_manager.py](/home/harold/workspace/music-player/infrastructure/database/sqlite_manager.py#L1051)
  - [plugin_sdk_runtime.py](/home/harold/workspace/music-player/system/plugins/plugin_sdk_runtime.py#L103)

## 6. 批量按 `cloud_file_id` 取曲目会覆盖多 provider 数据，导致队列补全和云盘播放串歌

- 位置：
  - `repositories/track_repository.py:134-143`
  - `services/playback/queue_service.py:176-235`
  - `services/playback/playback_service.py:777-805`
- 根因：
  - `get_by_cloud_file_ids()` 返回 `Dict[str, Track]`，键只有 `cloud_file_id`。
  - 当不同 provider 使用同一个 track id 时，后返回的记录覆盖先返回的记录。
  - 上层的队列元数据补全和云盘播放恢复都直接拿这个字典做映射。
- 影响：
  - 队列恢复时可能把 A 平台歌曲的标题/封面/track_id 套到 B 平台歌曲上。
  - 云盘/在线播放的元数据、收藏关联、后续下载状态都会被错误污染。
- 我这边的最小复现：
  - 插入 `qqmusic:42` 和 `netease:42` 后，`get_by_cloud_file_ids(['42'])` 只返回 1 条，解析出的标题是后插入的 `B`，provider 是 `netease`。
- 结论依据：
  - [track_repository.py](/home/harold/workspace/music-player/repositories/track_repository.py#L134)
  - [queue_service.py](/home/harold/workspace/music-player/services/playback/queue_service.py#L176)
  - [playback_service.py](/home/harold/workspace/music-player/services/playback/playback_service.py#L777)

## 7. mpv 后端清理依赖 `__del__`，正常关闭路径没有显式终止 backend，长测会触发进程级崩溃

- 位置：
  - `infrastructure/audio/audio_engine.py:115-127`
  - `infrastructure/audio/mpv_backend.py:248-259`
  - `services/playback/playback_service.py:981-987`
  - `ui/windows/main_window.py:2390-2415`
- 根因：
  - 真正会调用 `backend.cleanup()` 的地方只在 `PlayerEngine.__del__()`。
  - 正常关闭流程里只做了 `engine.stop()`、下载线程清理和 timer 停止，没有显式释放 mpv backend。
  - `python-mpv` 自己还有事件线程；如果对象没被及时析构，线程会残留到后续 Qt 事件循环。
- 已复现现象：
  - 执行 `uv run pytest tests/` 时，测试进行到 `tests/test_ui/test_cover_download_dialog.py::TestAlbumCoverDownloadDialog::test_album_single_item_mode` 附近，进程以 `139` 退出。
  - 崩溃栈里有多条 `mpv.py::_event_generator` 后台线程，说明不是单个断言失败，而是音频后端生命周期失控。
- 影响：
  - 长时间运行、频繁创建/销毁播放器对象、批量 UI 测试时，应用可能随机崩溃。
- 结论依据：
  - [audio_engine.py](/home/harold/workspace/music-player/infrastructure/audio/audio_engine.py#L115)
  - [mpv_backend.py](/home/harold/workspace/music-player/infrastructure/audio/mpv_backend.py#L248)
  - [playback_service.py](/home/harold/workspace/music-player/services/playback/playback_service.py#L981)
  - [main_window.py](/home/harold/workspace/music-player/ui/windows/main_window.py#L2390)
  - [test_cover_download_dialog.py](/home/harold/workspace/music-player/tests/test_ui/test_cover_download_dialog.py#L227)
